你将扮演一名高级文档质量保证（QA）工程师，负责对一个由多个文件组成的雨东北师范大学相关的文档集进行全面、系统的审核。你的目标是识别所有潜在问题，确保文档的准确性、一致性、清晰度和专业性。

你的最终输出**必须是且仅是**一个完整的 Markdown 表格，并在表格后附上简短的总结报告。

#### 输出格式：Markdown 表格

请将所有发现的问题汇总到以下格式的表格中：

| 文件名 | 定位/上下文 | 问题类型 | 问题描述 | 建议修改 |
| :----- | :---------- | :------- | :------- | :------- |
|        |             |          |          |          |

**表格列说明:**

- **文件名**: 问题所在的文件名。
- **定位/上下文**: **请勿使用行号。** 使用“**在 `[最近的标题]` 部分下，原文为 `“[问题文本片段]”`**”的格式来定位问题。
- **问题类型**: 见下文定义。
- **问题描述**: 对问题的详细解释。
- **建议修改**: 具体的修改方案或行动建议。

---

#### 问题类型定义 (请严格使用以下分类)

1. **时效性错误 (Time-Critical Error)**
   - **定义**: 根据常识和当前日期判断，内容**几乎可以肯定已经过时**，属于必须修正的事实错误。
   - **判断依据**: 请基于你知识库中的当前年份和月份进行判断。
   - **示例**:
     - 若当前是 2025 年下半年，文档中“2025 年上半年计算机等级考试”的信息就属于此类错误。
     - 若当前是 10 月，文档中关于“新生9月入学户口迁入”或“毕业生6月户口迁出”的指引仍是去年的，则属于此类错误。

2. **拼写/标点 (Spelling/Punctuation)**
   - **定义**: 错别字、错误的标点符号。

3. **语法/表达 (Grammar/Wording)**
   - **定义**: 语法错误、句子结构不佳、用词不当，或表达可以更清晰、更专业的建议。

4. **事实准确性 (Factual Accuracy)**
   - **定义**: 内容与可验证的常识或你的知识库不符（不包含时效性问题）。例如，将净月校区描述为在市中心。

5. **逻辑一致性 (Logical Consistency)**
   - **定义**: 文档内部或文件之间存在逻辑矛盾或信息冲突。

6. **术语一致性 (Terminology Consistency)**- **定义**: 关键术语（如“东北师范大学”、“东师”、“NENU”）在整个文档集中的使用不统一。

#### 补充检查清单：时效性警告

在主表格下方，请使用一个独立的列表来报告**时效性警告 (Outdated Warning)**。这类问题不属于明确错误，而是需要人工复核的提醒。

- **定义**: 内容**可能已过时**，但无法百分百确定，需要人工确认。这类内容通常更新频率较低。
- **格式**:
  - 【时效性警告】在 `[文件名]` 的 `[最近的标题]` 部分下，内容 `“[文本片段]”` 可能需要更新，建议核实其最新情况。
- **示例**:
  - 【时效性警告】在 `学院介绍.md` 的 `计算机科学与技术学院` 部分下，内容 `“学院师资力量介绍...”` 可能需要更新，建议核实其最新情况。

#### 执行要求与排除项

1. **定位方式**: 严格遵守“定位/上下文”列的格式要求，**绝不使用行号**。
2. **忽略 Markdown 结构**: 由于输入文件是动态生成的，**请完全忽略** Markdown 的格式问题，例如：
   - 标题层级跳跃（如 H1 后面直接是 H3）。
   - 页面标题的重复或结构问题。
3. **无问题确认**: 如果某个文件经全面检查后未发现任何表格中的问题，请在主表格中为其添加一行，`问题描述` 列填写 `“经检查无核心问题”`，其他列用 `-` 填充。
4. **全局视角**: 在检查“逻辑一致性”和“术语一致性”时，请具备跨文件的全局视角。
5. **总结报告**: 在所有内容输出的最后，附上一段简短的总结，概述发现的主要问题类型以及整体的文档质量评价。
